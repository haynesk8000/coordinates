## Fix Preset Selection Behavior

### Issue

Currently, when a user selects a coordinate system preset that is **already selected**, the application does not reposition or reconfigure the coordinate system. The preset change handler appears to ignore the selection because the preset ID has not changed.

As a result, the origin, axis orientation, labels, and other preset properties are **not reset** to their defined values if the user has previously modified the coordinate system.

### Required Behavior

Selecting a preset must **always** apply that preset's complete configuration, regardless of whether it is already the active preset.

Each preset selection should reset the coordinate system to the preset's defined state, including:

* Origin position
* Axis orientation
* Axis directions (including flips)
* Axis labels
* Variable names
* Any other preset-defined properties

### Implementation Requirements

* Do **not** treat selecting the currently active preset as a no-op.
* The preset application logic should execute every time a preset is selected.
* If the UI framework suppresses events when the selected value does not change, modify the implementation so that clicking or selecting a preset always invokes the preset application function.
* Separate the concept of **currently selected preset** from **applying a preset configuration**. Applying a preset should be an explicit action that can occur multiple times, even for the same preset.
* Ensure all dependent UI elements, equations, vectors, and visualizations update after the preset is reapplied.

### Acceptance Criteria

1. Select a preset.
2. Modify the origin and/or rotate or flip the axes.
3. Select the same preset again.
4. The coordinate system immediately returns to the preset's original configuration.
5. All equations, vectors, labels, and dependent UI components update correctly.
6. This behavior is consistent for every preset.
